# FlowBoard Security & RBAC Architecture Analysis

This document provides a highly extensive technical analysis of the Role-Based Access Control (RBAC) mechanisms, Multi-Tenancy segregation, and overall security posture implemented in the FlowBoard Project Management application.

---

## 1. Authentication & Identity Management
### 1.1 Core Authentication
FlowBoard utilizes `Flask-Login` for stateful session management. 
*   **Session State**: Every authenticated user session is managed securely via browser cookies with standard Flask cryptographic signing (`SECRET_KEY`).
*   **Route Protection**: Application endpoints are locked down using the `@login_required` decorator, preventing unauthenticated access to the main SPA, APIs, or data models.
*   **Credential Security**: User passwords are not stored in plaintext. They are salted and hashed utilizing `werkzeug.security.generate_password_hash` (typically using pbkdf2:sha256).

### 1.2 Multi-Tenant Organization Enforcement
FlowBoard operates on a multi-tenant workspace architecture. 
*   **Context Binding**: A global `@bp.before_app_request` hook acts as a security middleware. For every authenticated request, it verifies the existence of an `active_org_id` in the user's session.
*   **Orphan Prevention**: If a user attempts to bypass workspace selection or is not part of any organization, the middleware automatically interrupts the request lifecycle and forcefully redirects them to the workspace creation/join page.
*   **Data Isolation**: All major ORM queries (Projects, Tasks, Memberships) are strictly scoped against `organization_id`. This prevents cross-tenant data leakage (IDOR vulnerabilities), ensuring that an authenticated user in Workspace A cannot enumerate or modify records in Workspace B, even if they guess the sequential database IDs.

## 2. Role-Based Access Control (RBAC)
The application enforces a dual-layered RBAC model: **Workspace-Level** and **Project-Level**.

### 2.1 Workspace-Level Permissions (Organization)
The `Membership` model joins a `User` to an `Organization` with a specific `role`:
*   **ADMIN**: Possesses full read/write privileges over the entire workspace. Admins are the only users authorized to modify global workspace settings (name, logo), invite new members, remove existing members, or execute the destructive `delete_workspace` operation.
*   **MEMBER**: Standard access level. Members can view projects, be assigned to tasks, comment on issues, and create new projects/tasks within the workspace context, but cannot modify the workspace hierarchy itself.

### 2.2 Project-Level Permissions (The `@requires_project_manager` Decorator)
To enforce granular control over destructive project actions, FlowBoard implements a custom robust decorator: `app.utils.rbac.requires_project_manager`.

**How the Middleware Works:**
1.  **Parameter Extraction**: The decorator dynamically introspects the request context (`kwargs`) to extract identifiers like `project_id`, `task_id`, or `sprint_id`. 
2.  **Entity Resolution**: If only a `task_id` or `sprint_id` is present, it securely queries the database to resolve the parent `project_id`.
3.  **Authorization Matrix**: The user is granted "Project Manager" privileges ONLY if they meet at least one of the following hard conditions:
    *   **Project Lead**: Their `user.id` strictly matches the `project.lead_id`.
    *   **Project Creator**: Their `user.id` matches the `project.created_by_id`.
    *   **Workspace Admin Override**: Their workspace membership role is explicitly `'ADMIN'`.
4.  **Enforcement**: If none of these conditions are met, the request is aborted, an error flash message is generated, and the user is redirected away from the protected route.

**Protected Actions:**
This decorator is deliberately placed over sensitive, destructive, or highly administrative endpoints, including:
*   Project deletion.
*   Task deletion (and batch task deletion).
*   Sprint deletion or modification.
*   Modifying core project constraints.

## 3. General Security Posture & Vulnerability Mitigation

### 3.1 SQL Injection (SQLi) Prevention
FlowBoard completely avoids raw SQL string concatenation. All database interactions are strictly routed through **SQLAlchemy ORM**. The ORM utilizes parameterized queries under the hood, inherently neutralizing SQL injection vectors across all CRUD operations.

### 3.2 Cross-Site Request Forgery (CSRF) & Cross-Site Scripting (XSS)
*   **XSS Mitigation**: Jinja2 templating is used for frontend rendering. Jinja2 automatically escapes HTML tags and javascript payloads by default in variable outputs (e.g., `{{ user.username }}`). 
*   **Rich Text Safety**: For areas where HTML *is* rendered (like the Quill.js task descriptions generated by AI or edited by users), the payload is managed client-side by Quill's internal sanitizer, though server-side HTML sanitation (like `bleach`) is recommended for absolute strictness in production.

### 3.3 File Upload Security & Path Traversal Mitigation
When users upload attachments (e.g., task files, workspace logos):
*   **Filename Sanitization**: The system strictly enforces the `werkzeug.utils.secure_filename` utility. This strips out dangerous path manipulation characters (e.g., `../`, `..\\`, `/etc/passwd`) from the user's uploaded filename.
*   **Storage Isolation**: Uploaded files are written to specific, isolated static directories (`static/uploads/attachments/` or `static/uploads/logos/`), preventing attackers from overwriting critical application scripts or configuration files.

### 3.4 API Security (Insecure Direct Object Reference - IDOR)
While the database uses predictable sequential integers for primary keys, IDOR vulnerabilities are mitigated by ownership checks. For instance, when a user marks a notification as read via `/notifications/read/<int:id>`, the system verifies `if notification.user_id == current_user.id:` before committing the change. Similarly, task updates verify the task belongs to the user's active organization project before processing.

---

## Conclusion
FlowBoard implements a highly secure, defensively programmed architecture. By combining multi-tenant context enforcement, strict ORM parameterization, targeted RBAC decorators for critical write-paths, and standardized file sanitization, the platform successfully guards against the vast majority of OWASP Top 10 web vulnerabilities.
